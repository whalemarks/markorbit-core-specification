# CHANGELOG

All notable changes to the MarkOrbit Core Specification are documented in this file.

This project follows the principles defined in the MarkOrbit Constitution.

Architectural changes are intentionally conservative.

Implementation changes may occur frequently.

Architectural changes occur only after formal review.

---

# Versioning Policy

MarkOrbit follows Semantic Versioning at the architectural level.

```
MAJOR.MINOR.PATCH
```

## MAJOR

Breaking architectural evolution.

Examples:

- new architectural layer
- removal of a canonical object
- contract incompatibility
- constitutional changes

MAJOR versions should be rare.

---

## MINOR

Backward-compatible architectural expansion.

Examples:

- new specifications
- new capability domains
- additional workflow models
- new extension points

---

## PATCH

Documentation improvements.

Examples:

- terminology clarification
- examples
- diagrams
- editorial improvements
- typo corrections

PATCH versions never change architecture.

This is the first complete architectural specification of the MarkOrbit Professional Operating System.

Version 2.0 establishes a unified architectural language spanning professional knowledge, reasoning, execution, coordination, experience and organizational context.

The architecture is technology-independent and designed for long-term evolution.

## Introduced Universal Layer Pattern

Every professional layer now follows:

```text
Object

↓

Model

↓

Connection

↓

Product
```

This pattern becomes the universal architectural template for future specifications.

---

## Introduced Professional Value Flow

Established one universal value flow.

```text
Reality

↓

Facts

↓

Understanding

↓

Judgment

↓

Action

↓

Coordination

↓

Experience
```

All future architectural work should remain consistent with this progression.

---

## Introduced Published Product Architecture

All architectural layers now communicate through published Products.

Products provide:

- stable contracts
- versioning
- replaceable implementations
- implementation independence

---

## Introduced Workspace Architecture

Workspace is redefined.

Workspace is no longer considered a tenant.

Workspace is defined as:

> An independent operating context hosting one Professional Operating System.

---

## Introduced Federation

Organizations collaborate through federation.

Every Workspace remains autonomous.

Platform enables collaboration without shared ownership.
