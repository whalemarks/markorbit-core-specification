# Purpose

The Trademark Universe already exists.

The challenge is not to create it.

The challenge is to operate it.

MarkOrbit is designed as the Operating System of the Trademark Universe.

Its architecture transforms a complex professional ecosystem into a continuously evolving digital infrastructure.

This document defines that architecture.

It explains how the Trademark Universe is represented, understood, operated and delivered through MarkOrbit.

---

# Architecture Philosophy

MarkOrbit is not designed around applications.

It is not designed around databases.

It is not designed around Artificial Intelligence.

It is designed around reality.

Reality defines architecture.

Architecture enables operation.

Products deliver experience.

Technology serves architecture.

Architecture serves reality.

---

# Three Worlds

MarkOrbit consists of three connected worlds.

```
Reality World

↓

Operating World

↓

Application World
```

Each world owns a different responsibility.

Together they form the complete operating architecture.

---

# Reality World

Reality exists independently of MarkOrbit.

It includes everything that exists in the Trademark Universe.

Examples include:

- trademarks
- brands
- owners
- agencies
- attorneys
- customers
- trademark offices
- laws
- regulations
- documents
- legal events

Reality cannot be redesigned by software.

Reality is observed.

Reality is represented.

Reality is never owned by MarkOrbit.

---

# Operating World

The Operating World is MarkOrbit itself.

It transforms reality into reusable professional capability.

```
Reality

↓

Entity

↓

Data

↓

Brain

↓

Intelligence

↓

Capability

↓

Guide

↓

Workflow

↓

Workspace
```

This is the permanent operating system.

Everything inside MarkOrbit belongs to this world.

---

# Application World

Applications allow users to access the Operating World.

Examples include:

- MarkOrbit Lite
- MarkOrbit Enterprise
- MarkReg
- Marketplace
- API
- Mobile Applications

Applications evolve rapidly.

The Operating World remains stable.

Applications should never redefine the Operating World.

# Core Responsibilities

Every Core component owns exactly one primary responsibility.

| Component | Responsibility |
|------------|----------------|
| Entity | Represent |
| Data | Describe |
| Brain | Explain |
| Intelligence | Judge |
| Capability | Perform |
| Guide | Understand |
| Workflow | Execute |
| Workspace | Operate |

No responsibility should have multiple owners.

No component should perform another component's responsibility.

---

# Architectural Boundaries

The responsibilities of each component are intentionally separated.

## Entity

Represents reality.

Does not explain reality.

---

## Data

Describes facts.

Does not interpret facts.

---

## Brain

Explains facts.

Does not execute business.

---

## Intelligence

Produces professional judgment.

Does not execute work.

---

## Capability

Performs professional tasks.

Does not own business state.

---

## Guide

Understands intent.

Manages interaction context.

Does not execute business.

---

## Workflow

Executes business processes.

Owns execution state.

Does not perform professional reasoning.

---

## Workspace

Operates organizations.

Does not redefine MarkOrbit Core.

---

# Dependency Rules

Architectural dependencies always flow in one direction.

```
Reality World

↓

Operating World

↓

Application World

↓

Users
```

Lower layers may depend on higher layers.

Higher layers must never depend on lower layers.

Applications depend on the Operating World.

The Operating World never depends on applications.

---

# Shared Infrastructure

MarkOrbit Core provides shared infrastructure for every product.

Shared infrastructure includes:

- Entity Model
- Data Products
- Knowledge Products
- Intelligence Products
- Capability Catalog
- Guide
- Workflow

Shared infrastructure belongs to the ecosystem.

Not to individual products.

---

# Workspace

Workspace bridges shared infrastructure and organizational operation.

Workspace may configure:

- members
- permissions
- regional policies
- business rules
- local resources
- integrations

Workspace extends the Core.

Workspace never changes the Core.

---

# Architectural Laws

The following laws should always remain true.

### Reality First

Reality defines software.

Software never defines reality.

---

### Core First

Professional capability belongs to the Core.

Applications consume the Core.

Applications never redefine the Core.

---

### Guide Before Workflow

Business execution begins only after user intent has been sufficiently understood.

Guide prepares.

Workflow executes.

---

### Capability Before Workflow

Workflow organizes Capabilities.

Capability performs professional work.

Workflow never replaces Capability.

---

### Timeline Before Current State

History is permanent.

Current State is temporary.

Timeline is the foundation of truth.

---

### Reference Before Duplication

Entities should be referenced.

Knowledge should be referenced.

Capabilities should be referenced.

Duplication should always be avoided whenever possible.

---

# Evolution Strategy

Different architectural layers evolve at different speeds.

| Layer | Evolution |
|---------|-----------|
| Trademark Universe | Very Slow |
| MarkOrbit Core | Slow |
| Workspace | Moderate |
| Products | Fast |
| User Experience | Continuous |

The closer a layer is to reality, the slower it should change.

The closer a layer is to users, the faster it may evolve.
